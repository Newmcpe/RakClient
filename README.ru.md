# RakClient — асинхронный клиент SA-MP 0.3.7 на Rust

RakClient — это консольный клиент-бот для SA-MP 0.3.7, написанный на Rust с использованием
асинхронной среды Tokio. Он не открывает игру: вместо этого он сам говорит с сервером по сети —
ровно так, как это делал бы настоящий SA-MP. Протокол восстановлен из оригинального
`RakSAMP Lite.exe`, поэтому клиент подключается к живым серверам (в том числе Arizona RP) и доходит
до спавна.

> Если вы только начинаете — это нормально. Ниже всё по шагам.

## Что он умеет

- Подключается к серверу 0.3.7 и проходит весь путь от подключения до появления персонажа в игре:

  ```
  Отключён → Подключение → RakNet → Вход (ClientJoin)
  → В игре (InitGame) → Выбор класса → Запрос спавна → Заспавнен → синхронизация на ногах
  ```

- Реализует «RakNet 3.x» — надёжный/упорядоченный транспорт поверх UDP и побайтовый шифр пакетов.
- Принимает и отправляет чат (в кодировке cp1251, как на русских серверах).
- Имеет слой скриптов на **Lua (Luau)**, повторяющий привычный API MoonLoader / SAMP.Lua —
  можно перехватывать любые RPC и пакеты прямо из Lua.

Конечная цель проекта — мост чата «Android ↔ ПК» для серверов Arizona.

## С чего начать

Нужен установленный [Rust](https://rustup.rs/) (команды `cargo` и `rustc`).

Собрать весь проект:

```sh
cargo build --workspace
```

Запустить клиент (имя бинарника — `rakclient`, а не `app`):

```sh
cargo run -p app -- --server <хост:порт> --nick ВашНик
```

Для серверов **Arizona** эмуляция (CEF/валидация) живёт в Lua, а не в Rust. Подгрузите папку с
лаунчером — тайминг отправки валидации задаёт сам скрипт через `wait()`:

```sh
cargo run -p app -- --server bumblebee.arizona-rp.com:7777 --nick ВашНик \
  --scripts-dir example_scripts
```

Хотите видеть подробный лог — поставьте переменную окружения `RUST_LOG`:

```sh
RUST_LOG=info cargo run -p app -- --server <хост:порт> --nick ВашНик
# или raknet::transport=trace — чтобы видеть каждый пакет в виде hex
```

Скрипты подгружаются автоматически: все файлы `*.lua` / `*.luau` из папки скриптов (по умолчанию
`scripts/`, переопределяется флагом `--scripts-dir`):

```sh
cargo run -p app -- --server <хост:порт> --nick ВашНик --scripts-dir example_scripts
```

## Про пароли

- `--password` — пароль **самого сервера RakNet** (на Arizona его нет, оставьте пустым). Если
  положить сюда пароль аккаунта, сервер отклонит подключение (`ID_INVALID_PASSWORD`).
- Пароль **аккаунта** (диалог авторизации на сервере) ядро на Rust больше не вводит — этим занимается
  ваш Lua-скрипт для Arizona: он видит диалог через `samp.events.onShowDialog` и отвечает через
  `sampSendDialogResponse`.

## Как устроен проект

Проект разбит на крейты (пакеты Rust). Каждый отвечает за один слой, зависимости идут только вниз:

| Крейт | За что отвечает |
| --- | --- |
| `samp-proto` | Чистые кодеки: `BitStream`, идентификаторы пакетов/RPC, типизированная (де)сериализация. Без сети. |
| `raknet` | Транспорт RakNet 3.x: шифр, слой надёжности, асинхронный `RakPeer`. |
| `samp-client` | Машина состояний подключения и высокоуровневый `Client`. |
| `samp-script` | Хост Lua (Luau): нативные `bitStream` + `registerHandler` и типизированный порт `samp.events`. |
| `app` | Бинарник `rakclient`: чтение настроек, логирование, запуск клиента до спавна. |
| `test-support` | Только для тестов: моковый SA-MP сервер на localhost. |

## Проверка перед коммитом

В проекте есть «ворота качества» — они должны проходить зелёными:

```sh
cargo fmt --all --check
cargo clippy --all-targets --all-features -- -D warnings
cargo test --workspace
```

## Откуда взялся протокол

Формат пакетов восстановлен из оригинального бинарника: таблица RPC, поведение `BitStream`,
рукопожатие `ClientJoin` (`version = 4057`, `challengeResponse = cookie ^ 0xFD9`), раскладка
синхронизации «на ногах» и побайтовый шифр с таблицей замен на 256 байт. Эти значения не угадываются —
они выверены по бинарю, поэтому менять их можно только вместе с golden-vector тестом.
